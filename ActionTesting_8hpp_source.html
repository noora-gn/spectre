64 /*!
65  * \ingroup TestingFrameworkGroup
66  * \brief Structures used for mocking the parallel components framework in order
67  * to test actions.
68  *
69  * The ActionTesting framework is designed to mock the parallel components so
70  * that actions and sequences of actions can be tested in a controlled
71  * environment that behaves effectively identical to the actual parallel
72  * environment.
73  *
74  * ### The basics
75  *
76  * The action testing framework (ATF) works essentially identically to the
77  * parallel infrastructure. A metavariables must be supplied which must at least
78  * list the components used (`using component_list = tmpl::list<>`) and the
79  * phases (`enum class Phase {}`). As a simple example, let's look at the test
80  * for the `Parallel::Actions::TerminatePhase` action.
81  *
82  * The `Metavariables` is given by:
83  *
84  * \snippet Test_TerminatePhase.cpp metavariables
85  *
86  * The component list in this case just contains a single component that is
87  * described below. The `Phase` enum contains the standard `Initialization` and
88  * `Exit` phases. A `Testing` phase is also added, but the name of the phase has
89  * no special meaning. Multiple phases with different names could instead be
90  * used.
91  *
92  * The component is templated on the metavariables, which, while not always
93  * necessary, eliminates some compilation issues that may arise otherwise. The
94  * component for the `TerminatePhase` test is given by:
95  *
96  * \snippet Test_TerminatePhase.cpp component
97  *
98  * Just like with the standard parallel code, a `metavariables` type alias must
99  * be present. The chare type is always `ActionTesting::MockArrayChare` since
100  * singletons may be treated during action testing as just a one-element array
101  * component. The `index_type` must be whatever the actions will use to index
102  * the array. In the case of a singleton `int` is recommended. Finally, the
103  * `phase_dependent_action_list` must be specified just as in the cases for
104  * parallel executables. In this case only the `Testing` phase is used and only
105  * has the `TerminatePhase` action listed.
106  *
107  * The `SPECTRE_TEST_CASE` is
108  *
109  * \snippet Test_TerminatePhase.cpp test case
110  *
111  * The type alias `component = Component<Metavariables>` is just used to reduce
112  * the amount of typing needed. The ATF provides a
113  * `ActionTesting::MockRuntimeSystem` class, which is the code that takes the
114  * place of the parallel runtime system (RTS) (e.g. Charm++) that manages the
115  * actions and components. Components are added to the RTS using the
116  * `ActionTesting::emplace_component()` function, which takes the runner by
117  * `not_null`, the array index of the component being inserted (`0` in the above
118  * example, but this is arbitrary), and optionally a parameter pack of
119  * additional arguments that are forwarded to the constructor of the component.
120  * These additional arguments are how input options (set in the
121  * `initialization_tags` type alias of the parallel component) get passed to
122  * parallel components.
123  *
124  * With the ATF, everything is controlled explicitly, providing the ability to
125  * perform full introspection into the state of the RTS at virtually any point
126  * during the execution. This means that the phase is not automatically advanced
127  * like it is in a parallel executable, but instead must be advanced by calling
128  * the `ActionTesting::set_phase` function. While testing the `TerminatePhase`
129  * action we are only interesting in checking whether the algorithm has set the
130  * `terminate` flag. This is done using the `ActionTesting::get_terminate()`
131  * function and is done for each distributed object (i.e. per component). The
132  * next action in the action list for the current phase is invoked by using the
133  * `ActionTesting::next_action()` function, which takes as arguments the runner
134  * by `not_null` and the array index of the distributed object to invoke the
135  * next action on.
136  *
137  * ### InitializeDataBox and action introspection
138  *
139  * Having covered the very basics of ATF let us look at the test for the
140  * `Actions::Goto` action. We will introduce the functionality
141  * `InitializeDataBox`,`force_next_action_to_be`, `get_next_action_index`, and
142  * `get_databox_tag`. The `Goto<the_label_class>` action changes the next action
143  * in the algorithm to be the `Actions::Label<the_label_class>`. For this test
144  * the Metavariables is:
145  *
146  * \snippet Test_Goto.cpp metavariables
147  *
148  * You can see that this time there are two test phases, `TestGoto` and
149  * `TestRepeatUntil`.
150  *
151  * The component for this case is quite a bit more complicated so we will go
152  * through it in detail. It is given by
153  *
154  * \snippet Test_Goto.cpp component
155  *
156  * Just as before there are `metavariables`, `chare_type`, and `array_index`
157  * type aliases. The `repeat_until_phase_action_list` is the list of iterable
158  * actions that are called during the `TestRepeatUntil` phase. The
159  * `Initialization` phase action list now has the action
160  * `ActionTesting::InitializeDataBox`, which takes simple tags and compute tags
161  * that will be added to the DataBox in the initial phase. How the values for
162  * the simple tags are set will be made clear below when we discuss the
163  * `ActionTesting::emplace_component_and_initialize()` function. The action
164  * lists for the `TestGoto` and `TestRepeatUntil` phases are fairly simple and
165  * not the focus of this discussion.
166  *
167  * Having discussed the metavariables and component, let us now look at the test
168  * case.
169  *
170  * \snippet Test_Goto.cpp test case
171  *
172  * Just as for the `TerminatePhase` test we have some type aliases to reduce the
173  * amount of typing needed and to make the test easier to read. The runner is
174  * again created with the default constructor. However, the component is now
175  * inserted using the `ActionTesting::emplace_component_and_initialize()`
176  * function. The third argument to
177  * `ActionTesting::emplace_component_and_initialize()` is a
178  * `tuples::TaggedTuple` of the simple tags, which is to be populated with the
179  * initial values for the component. Note that there is no call to
180  * `ActionTesting::set_phase (&runner, Metavariables::Phase::Initialization)`:
181  * this and the required action invocation is handled internally by
182  * `ActionTesting::emplace_component_and_initialize()`.
183  *
184  * Once the phase is set the next action to be executed is set to be
185  * `Actions::Label<Label1>` by calling the
186  * `MockRuntimeSystem::force_next_action_to_be()` member function of the
187  * runner. The argument to the function is the array index of the component for
188  * which to set the next action. After the `Label<Label1>` action is invoked we
189  * check that the next action is the fourth (index `3` because we use zero-based
190  * indexing) by calling the `MockRuntimeSystem::get_next_action_index()` member
191  * function. For clarity, the indices of the actions in the `Phase::TestGoto`
192  * phase are:
193  * - 0: `Actions::Goto<Label1>`
194  * - 1: `Actions::Label<Label2>`
195  * - 2: `Actions::Label<Label1>`
196  * - 3: `Actions::Goto<Label2>`

210  * There is also a `gsl::not_null` overload of `ActionTesting::get_databox()`

217  * ### Stub actions and invoking simple and threaded actions
218  *
219  * A simple action can be invoked on a distributed object or component using
220  * the `ActionTesting::simple_action()` function:
221  *
222  * \snippet Test_ActionTesting.cpp invoke simple action
223  *
224  * A threaded action can be invoked on a distributed object or component using
225  * the `ActionTesting::threaded_action()` function:
226  *
227  * \snippet Test_ActionTesting.cpp invoke threaded action
228  *
229  * Sometimes an individual action calls other actions but we still want to be
230  * able to test the action and its effects in isolation. To this end the ATF
231  * supports replacing calls to actions with calls to other actions. For example,
232  * instead of calling `simple_action_a` we want to call `simple_action_a_mock`
233  * which just verifies the data received is correct and sets a flag in the
234  * DataBox that it was invoked. This can be done by setting the type aliases
235  * `replace_these_simple_actions` and `with_these_simple_actions` in the
236  * parallel component definition as follows:
237  *
238  * \snippet Test_ActionTesting.cpp simple action replace
239  *
240  * The length of the two type lists must be the same and the Nth action in
241  * `replace_these_simple_actions` gets replaced by the Nth action in
242  * `with_these_simple_actions`. Note that `simple_action_a_mock` will get
243  * invoked in any context that `simple_action_a` is called to be invoked. The
244  * same feature also exists for threaded actions:
245  *
246  * \snippet Test_ActionTesting.cpp threaded action replace
247  *
248  * Furthermore, simple actions invoked from an action are not run immediately.
249  * Instead, they are queued so that order randomization and introspection may

255  * The `ActionTesting::invoke_queued_simple_action()` invokes the next queued
256  * simple action:
257  *
258  * \snippet Test_ActionTesting.cpp invoke queued simple action
259  *
260  * Note that the same functionality exists for threaded actions. The functions
261  * are `ActionTesting::is_threaded_action_queue_empty()`, and
262  * `ActionTesting::invoke_queued_threaded_action()`.

542 /// MockDistributedObject mocks the AlgorithmImpl class. It should not be
543 /// considered as part of the user interface.
544 ///
545 /// The `MockDistributedObject` represents a single chare or distributed object
546 /// on a supercomputer which can have methods invoked on it. This class is a
547 /// modified implementation of `AlgorithmImpl` and so some of the code is shared
548 /// between the two. The main difference is that `MockDistributedObject` has
549 /// support for introspection. For example, it is possible to check how many
550 /// simple actions are queued, to look at the inboxes, etc. Another key
551 /// difference is that `MockDistributedObject` runs only one action in the
552 /// action list at a time. This is done in order to provide opportunity for
553 /// introspection and checking statements before and after actions are invoked.
554 template <typename Component>
555 class MockDistributedObject {
556  private:
557  class InvokeActionBase {
558  public:
